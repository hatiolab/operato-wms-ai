## 13:00 — OMS 피킹 유형(pick_type) 구조 개편

### 작업 내용
- ShipmentWave/ShipmentOrder의 `pick_method`, `pick_type` 구조 재설계
- **기존 문제**: pick_type(INDIVIDUAL/TOTAL/ZONE) + pick_method(WCS/PAPER/INSPECT/PICK) 조합이 복잡하고 명확하지 않음
- **개편 방향**: pick_type만 유지하고 wcs_flag(Boolean)로 분리
  - pick_type: TOTAL(토털 피킹) / ZONE(존별 피킹) / INDIVIDUAL(개별 피킹)
  - wcs_flag: WCS 자동 처리 여부
  - insp_flag: 검수 필요 여부 (기존 유지)

### 변경/생성 파일

**1. ShipmentWave 엔티티**
- `src/main/java/operato/wms/oms/entity/ShipmentWave.java`
  - ❌ 제거: `pick_method` 필드
  - ✅ 추가: `wcs_flag` (Boolean) — WCS 위임 여부
  - ✅ 수정: `pick_type` 값 변경 (INDIVIDUAL/TOTAL/ZONE → TOTAL/ZONE/INDIVIDUAL)

**2. ShipmentOrder 엔티티**
- `src/main/java/operato/wms/oms/entity/ShipmentOrder.java`
  - ❌ 제거: `pick_type` 필드 (정규화 — wave_no로 조회 가능)
  - ✅ 수정: 인덱스 `ix_shipment_orders_5`에서 pick_type 제거

**3. 이벤트 구조 변경**
- `src/main/java/operato/wms/common/event/WaveReleasedEvent.java`
  - ❌ 제거: `pickMethod` 필드
  - ✅ 추가: `wcsFlag`, `inspFlag` 필드
- `src/main/java/operato/wms/fulfillment/service/FulfillmentEventListener.java`
  - wcsFlag 기반 pick_method 결정: true → "WCS", false → "PICK"

**4. 웨이브 확정 로직**
- `src/main/java/operato/wms/oms/service/OmsTransactionService.java`
  - 웨이브 확정 시 주문 업데이트: 상태만 변경 (pick_type, carrier_cd 업데이트 제거)
  - carrier_cd는 조회 시 `COALESCE(order.carrier_cd, wave.carrier_cd)` 로직으로 처리

**5. 설계 문서**
- `docs/design/oms-table-design.md`
  - shipment_waves: pick_method → wcs_flag
  - shipment_orders: pick_type 컬럼 제거
- `docs/design/oms-screen-design.md`
  - 웨이브 목록: pick_method → wcs_flag, insp_flag 체크박스
  - 웨이브 상세: "피킹방식: PICK" → "WCS위임: Y  검수: Y"
  - 자동 웨이브 생성 팝업: "피킹 방식" 제거, WCS/검수 체크박스 추가

**6. 용어 등록** (skill: `/add_term`, `/clear_frontend_cache`)
- terminologies 테이블에 `wcs_flag` 용어 추가 (ko/en/ja/zh, 6개 도메인 × 4개 언어 = 24건)

### 주요 결정 또는 발견 사항

**1. 설계 원칙: 직교성(Orthogonality)**
- 피킹 전략(pick_type)과 실행 방법(wcs_flag)을 독립적으로 분리
- 조합 가능: TOTAL + WCS, ZONE + PICK, INDIVIDUAL + WCS 등

**2. 데이터 정규화**

- carrier_cd 업데이트 제거: 웨이브 확정 취소 시 복원 불가 문제 해결
- 조회 시 `COALESCE(order.carrier_cd, wave.carrier_cd)` 패턴 사용

**3. 이벤트 기반 아키텍처 유지**

- FulfillmentEventListener에서 wcsFlag 기반 pick_method 결정 (WCS/PICK)
- PickingTask는 기존 pick_method 필드 유지 (Fulfillment 모듈 독립성)

**4. 웨이브 확정 취소 안정성**

  - GET `/rest/ful_trx/picking_tasks/done` — 작업 완료 목록
  - GET `/rest/ful_trx/picking_tasks/{id}` — 상세 조회
- `src/main/java/operato/wms/fulfillment/service/FulfillmentPickingService.java`
  - `searchTodoPickingTasks()` — CREATED/IN_PROGRESS 상태 조회
  - `searchDonePickingTasks()` — COMPLETED 상태 조회
  - `getPickingTask(String id)` — 단건 상세 조회

**5. API 경로 수정**
- `frontend/packages/operato-wes/client/pages/fulfillment/fulfillment-picking-pc.js`
  - API 경로 일괄 변경: `fulfillment_trx` → `ful_trx` (6개 위치)

### 주요 결정 또는 발견 사항

**1. API 구조 개선**
- 기존 API 경로가 `/rest/fulfillment_trx`로 예상했으나 실제는 `/rest/ful_trx`
- 목록 조회 API가 없어 todo/done 2개 엔드포인트 추가
- 상세 조회 API 추가로 화면에서 작업 정보 조회 가능

**2. 화면 구조**
- **좌측 패널**: 피킹 태스크 목록 (검색, 필터링, 우선순위 정렬)
- **우측 패널**: 3가지 모드
  - empty: 작업 선택 전 안내
  - work: 아이템별 피킹 진행 (바코드 스캔, 수량 입력, 진행률)
  - complete: 작업 완료 통계 (소요 시간, SKU 수, 총 수량)

**3. 작업 흐름**
1. 좌측에서 작업 선택 → 자동으로 START 상태로 전환
2. 우측에서 SKU별 바코드 스캔 → 수량 확인 → 피킹 확인
3. 모든 아이템 완료 → COMPLETED 상태 + 완료 통계 표시
4. 다음 작업 시작 또는 목록으로 복귀

**4. 상태 관리**
- PickingTask 상태: CREATED (대기) → IN_PROGRESS (진행) → COMPLETED (완료)
- PickingTaskItem 상태: WAIT → RUN → PICKED
- 프론트엔드에서 자동으로 다음 미완료 아이템으로 포커스 이동
